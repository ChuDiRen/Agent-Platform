# 需求评估技能

## 评估流程 (严格按顺序)

### 第一步: 提取需求条目
调用 `extract_requirements` 从文档中识别:
- 用户故事 (以"作为...我想...以便..."格式)
- 业务规则 (以"必须/应当/不允许"等约束词标记)
- 接口定义 (URL/Method/参数/响应)
- 界面说明 (页面/按钮/输入框)
- 流程描述 (步骤/状态/跳转)

### 第二步: 完整性检查
对每个维度调用 `check_completeness`:
1. **流程完整性** — 是否有跳转目标、状态流转、操作反馈
2. **输入校验** — 是否定义格式、长度、类型、必填项
3. **安全规则** — 是否覆盖密码策略、会话管理、鉴权
4. **异常处理** — 是否说明错误、超时、重试、并发
5. **权限控制** — 是否定义角色、权限边界

### 第三步: 分类与评级
对每个发现调用 `categorize_finding` 和 `assess_severity`。

## 严重程度标准

| 等级 | 标准 | 示例 |
|------|------|------|
| HIGH | 影响功能正确性或系统安全 | 无鉴权说明、密码明文、缺少支付校验 |
| MEDIUM | 影响测试覆盖或用户体验 | 缺少错误提示、无输入校验、无异常处理 |
| LOW | 可优化的交互细节 | 按钮文案、加载动画、提示语措辞 |

## 输出模板

```json
[
  {
    "id": "missing-auth-flow",
    "title": "缺少鉴权流程说明",
    "description": "文档未说明登录态校验机制、Token 过期处理和未授权跳转逻辑。建议补充 JWT/Session 鉴权流程图和各环节的错误处理。",
    "severity": "high",
    "category": "安全规则"
  }
]
```

## 原则
- 只报告真正的问题，不凑数
- 每个问题必须给出具体补充建议
- 避免重复: 同一类别下合并相似问题
- 描述要具体，引用文档中的具体内容
